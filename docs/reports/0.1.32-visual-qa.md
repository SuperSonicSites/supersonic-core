## 7. Findings By Severity

No critical, high, or low findings were confirmed this run. Two medium and three nit findings were verified against source.

### Medium

| # | Pattern | Finding | Fix |
| --- | --- | --- | --- |
| M1 | section-stat-band | Columns stack at tablet (768px). The `core/columns` block (source line 12) has no `isStackedOnMobile:false` attribute, so WordPress applies its default `max-width:781px` stacking rule and the 768px tablet viewport collapses the three-up stat band to a single column. Content stays readable but the intended horizontal "proof band" layout is lost at tablet. | Add `"isStackedOnMobile":false` to the `wp:columns` block (renders `is-not-stacked-on-mobile`). Note: the same default affects sibling multi-column patterns (feature-grid, card-row, pricing-cards, service-cards, process-steps, testimonial-grid, comparison, footer-simple); if accepted as a defect, treat it as a systemic decision, not a stat-band one-off. |
| M2 | section-pricing-cards | Price and feature rows do not align horizontally across the three desktop cards. The Growth (featured) card's inner group begins with an extra `Popular` paragraph (source lines 52–54) above its H3, with no compensating spacer/min-height on the Starter and Scale cards, so the Growth H3 and `$0` price sit one text line lower than the other two — breaking the at-a-glance price comparison. | Add a matching empty/spacer paragraph (or a `min-height` on the card-header area) to the Starter and Scale inner groups so all three card-content flows start at the same vertical position. |

### Nit

| # | Pattern | Finding | Fix |
| --- | --- | --- | --- |
| N1 | hero-simple | Both CTAs use placeholder `href="#"` (source lines 27 and 31). Expected for a self-described starter pattern; not a token violation. | Authors must set real destination URLs before this pattern is used on a live page. |
| N2 | hero-centered | Primary `Get started` CTA uses placeholder `href="#"` (source line 29). Intentional starter-skeleton convention; not a token violation. | Replace with a real URL before any production page uses this pattern. |
| N3 | section-image-text | Pattern description header says "action buttons" (plural) but only a single primary CTA is authored in the `wp:buttons` container. Documentation/copy imprecision with no functional, token, accessibility, or layout impact. | Either reword the description to singular or add a secondary button; the `wp:buttons` container natively supports adding more. |

## 8. Considered And Dismissed

Listed so future runs do not re-flag them. Each was checked against pattern source and/or the cited screenshot and found not to be a source-grounded defect.

| Finding | Why dismissed |
| --- | --- |
| hero-simple: mobile CTAs lack `flex-wrap`, may feel cramped at 390px | False premise. `core/buttons` defaults to `flex-wrap: wrap` when `flexWrap` is omitted, so the buttons already reflow; disabling wrap requires explicit `"flexWrap":"nowrap"` (used elsewhere, not here). No theme CSS overrides it. The proposed fix is a no-op; two short labels fitting on one line is not a defect. |
| hero-centered: placeholder `href="#"` on secondary outline CTA | Intentional skeleton-pattern convention. `href="#"` is the universal placeholder across all 18 button instances in 10 pattern files; it is valid, not a broken/dangling link, and is filled in during page assembly. The genuine pre-production reminder is already captured by N2 for the primary button. |
| contact-simple: no primary CTA anchor in the section | Intentional minimal variant (the finding concedes the title signals intent). The section already has working `mailto:`/`tel:` actions, and a conversion-oriented sibling (`contact-form-simple`) exists per the modularity rule. CTA-presence is a page/view-level rule owned by layout-review, not a per-section visual-QA defect. |
| contact-simple: tablet paragraph warm tint in screenshot | Screenshot artifact. The intro paragraph has no color attribute and resolves to the global `contrast` token `#111111`; the palette has no warm/orange token, and no media-query color CSS exists. Warm tint appears only in the tablet PNG (near-black at desktop/mobile) → Playwright capture color-profile artifact. |
| section-stat-band: label text warm/orange cast on mobile | Screenshot artifact. Labels set only `align`+`small` and inherit `contrast` `#111111`; the entire palette is neutral grayscale + one blue accent, with `color.custom`/`defaultPalette` false. No CSS tints small paragraphs and no media query changes color. Identical markup renders neutral at desktop/tablet → gamma/rendering artifact. |
| section-pricing-cards: tablet subtitle renders in accent color | Refuted by source and pixels. The subtitle has no `textColor` and inherits `contrast` `#111111`; accent is applied only via `elements.link` and header-scoped nav CSS, neither of which can reach this (link-free) paragraph. The cited tablet PNG actually shows the subtitle near-black; the only blue elements are the intentionally accent `Popular` badge, the Growth border, and the CTAs. |
| section-pricing-cards: Growth card feature list inherits accent at mobile | Refuted by source. The Growth list inherits `contrast` from its `has-contrast-color` parent group, identical to Starter/Scale. The accent `Popular` paragraph is a sibling, not an ancestor — CSS `color` does not inherit sibling→sibling. No adjacent/sibling selector and no mobile media query exists to tint it. Apparent blue is a screenshot misread. |
| section-image-text: body paragraph renders orange/amber on tablet and mobile | Screenshot artifact. The paragraph has no `textColor` and inherits `contrast` `#111111` at every breakpoint; fluid typography changes size only, and the sole `max-width:781px` rule only changes flex `order`. No warm/orange value exists anywhere in the theme — even an (impossible) accent leak would render blue, not orange. |
| section-image-text: placeholder image empty `alt=""` with no editor reminder | Not a defect. Empty `alt` on a decorative placeholder is WCAG-correct (the finding concedes this). It is a native `core/image` block, so the editor surfaces an Alt-text field automatically on replace; the empty-alt convention is consistent across all four media patterns. The requested in-pattern nudge already exists via the block UI and documented media-slot ownership. |
| section-text-image: body paragraph renders orange/red at tablet | Screenshot artifact. Same mechanism as section-image-text: paragraph inherits `contrast` `#111111` at all breakpoints; the hypothesized `.wp-block-column p` / `.supersonic-media-split__content p` selectors do not exist; the only tablet media query sets `order` only; the palette contains no orange/red/brown token. Tablet-only warm tint with no source path is a capture artifact. |
| section-text-image: placeholder image empty `alt=""` with no editor reminder | Not a defect. The placeholder is a decorative SVG that self-describes via `role="img" aria-label="Editable image placeholder"`; empty `alt` is WCAG-correct. Same intentional convention across all four media patterns; the existing "replace before approval" comment plus the native editor Alt field cover the swap workflow. |

## 9. Remaining Risks / Verify Visually

- **Coverage gap.** 20 of 28 registered patterns were not captured this run. Their spacing, overflow, mobile stacking, and contrast are not visually confirmed for 0.1.32. Recommend capturing the remaining patterns (per-pattern QA pages, dry-run + approval) before a full release sign-off.
- **Tablet column stacking (M1) is potentially systemic.** The default `max-width:781px` core stacking applies to every multi-column section pattern, not just `section-stat-band`. Decide once whether multi-up bands should hold their layout at 768px tablet (`isStackedOnMobile:false`) and apply consistently.
- **Pricing card alignment (M2).** Until a spacer/min-height is added, the featured-card offset will recur on any page using `section-pricing-cards`.
- **Placeholder `href="#"`** on hero-simple and hero-centered CTAs (and the section CTAs) must be wired to real destinations before any page using these patterns goes live.
- **Manual-only gaps not exercised this run:**
  - Token editability in the WordPress block editor (spacing-preset, font-size, color picker, button-style toggles) was not verified in a live editor session for any captured pattern — confirm no block-validation errors on edit.
  - Button hover/focus-ring appearance (cta-band, hero-*, section CTAs) was not captured; no custom JS interaction exists, but visual focus state is unverified.
  - Real image replacement with meaningful alt text for the media patterns (section-image-text, section-text-image) is editor-dependent and unconfirmed.
  - Accent-blue link/badge contrast on white (`#1f6feb`, e.g. contact-simple links, pricing `Popular` badge) is borderline for WCAG AA on normal text and merits a dedicated accessibility pass rather than a visual judgment.
