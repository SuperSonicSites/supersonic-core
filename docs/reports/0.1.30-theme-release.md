# Theme 0.1.30 — Release Record

**Type:** theme-only release (plugin `supersonic-site-core` stays at 0.1.8).
**Date prepared:** 2026-06-02.
**Readiness:** GO once cut via the release Action (see Release procedure). A
multi-agent release-readiness review initially returned NO-GO on one
release-cutting blocker (version pre-bump vs the Action's self-bump); that fix is
applied (source reset to 0.1.29 so the Action owns the 0.1.30 bump).

## Theme upgrade surface (what staging gets after deploy)

These are the only runtime/presentation changes site owners receive when the
theme updates to 0.1.30:

- **Stacked-section seam fix (`SPACE-7`).** Adjacent full-width section bands now
  butt together — the default block-gap no longer leaves a base-colored seam
  between two colored bands. Vertical rhythm still comes from each section's
  `section-*` padding. (`wp-content/themes/supersonic-site-theme/assets/css/patterns.css`)
- **New approved gradient presets:** `surface-rise`, `accent-veil`, `muted-soft`
  (custom gradients stay disabled).
- **New approved shadow preset:** `strong` (`0 24px 56px rgb(17 17 17 / 0.16)`),
  alongside `soft` and `medium`.

No templates, patterns, or existing tokens changed behavior; existing patterns do
not use the new presets, so their rendering is unchanged.

## Framework additions (repo tooling/docs — no theme runtime impact)

The bulk of the changeset is the new AI page-layout system, which does not ship
in the theme zip:

- **Skills/commands:** `layout-architect` (compose a page) and `layout-review`
  (grade it across 16 rule-ID dimensions, feed fixes back); `/layout-page` and
  `/layout-review`; CLAUDE.md routing rows.
- **Standard:** `docs/layout-standard.md` — sourced 2026 rule catalog, page
  archetypes, alternating-background rhythm (`COLOR-6/7/8/9`), `SPACE-7` seam
  rule, control-exposure (`FW-7`), placeholder spec.
- **Validators:** `validate-framework.mjs` gradient/`strong` token checks +
  gradient backgrounds counted for the readable-text contract;
  `validate-agent-quality.mjs` fail-closed guard so a layout review cannot claim
  strong/approved while proof is partial/pending.
- **Tooling:** `staging-rest.mjs` QA-page create/update from composed `--content`
  payloads; `make-sample-service-payload.mjs` fixture generator.
- **Forward-tests (`docs/layouts/`):** a Service-archetype page and a
  Home-archetype plumbing page, each with blueprint, composed markup, payload,
  review, and live staging screenshots.

## Approval record

- **Design-token expansion approved by the site owner** (Human Approval Gates:
  "changing global design tokens", "adding or changing shadow presets"). The
  owner explicitly opted to expand the enabled control set with the three gradient
  presets and the `strong` shadow during this work. Values are on-token and
  on-brand (no arbitrary CSS).

## Release procedure

1. **Done:** source reset to 0.1.29 (`3b444fa`) so the release Action owns the
   0.1.30 bump (it commits "Release theme v0.1.30" + tags `theme-v0.1.30`).
2. **Owner action — cut the release:** dispatch the "Release theme to staging"
   GitHub Action (Actions → Run workflow) with version `0.1.30`, from `main`. It
   lints, bumps 0.1.29 → 0.1.30, packages, validates, publishes a SHA-256-verified
   GitHub Release (`theme-v0.1.30`), and pokes staging to pull it. **Production is
   never touched.**
3. **After deploy:** re-stage and reconfirm the two forward-test pages so their
   reviews move from `acceptable-pending-staging` / `needs-revision` to
   `approved` (the `SPACE-7` seam is only gone once 0.1.30 is live on staging).
